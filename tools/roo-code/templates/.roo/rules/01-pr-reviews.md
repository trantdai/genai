# Pull Request Review Standards

## 🚨 PR Review Task Splitting (MANDATORY)

### Context Window Risk
PR reviews are **HIGH RISK** for context window errors. **ALWAYS split when:**
- [ ] >5 files changed → Split by file groups (3-5 files per phase)
- [ ] >500 lines changed → Split by review aspect (security, quality, testing)
- [ ] >3 review categories → One category per response
- [ ] Complex changes → Split by logical component

### Before Starting PR Review
- [ ] Count files changed (if >5, **MUST SPLIT**)
- [ ] Estimate lines changed (if >500, **MUST SPLIT**)
- [ ] Identify logical components (if >3, **MUST SPLIT**)
- [ ] Create explicit phase plan
- [ ] Share plan with user for approval
- [ ] Execute ONE phase at a time

### 🎨 Formatting Requirements (MANDATORY)
**EVERY PR review comment MUST:**
- [ ] **Start with**: `# 🚀 This PR review was conducted by AI 🤖`
- [ ] **End with**: `Reminder: This PR comment was generated by AI 🤖. Not everything may be accurate - please verify!`

**These are NON-NEGOTIABLE formatting requirements for ALL PR reviews.**

### Splitting Strategies
**By File Groups**: Phase 1: core files (3-5) → Phase 2: tests → Phase 3: config → Phase 4: summary
**By Aspect**: Phase 1: security → Phase 2: quality → Phase 3: testing → Phase 4: summary
**By Component**: Phase 1: auth module → Phase 2: API → Phase 3: frontend → Phase 4: summary

**Details**: See [Task Management](./11-task-management.md#pr-review-splitting)

## General - MCP Integration
When asked to review a Pull Request on GitHub.com, always attempt to use the MCP Server, "github".

If the GitHub MCP Server cannot be connected to, do not proceed with the PR review. Do not attempt to review open files as a fallback - throw an error and inform the user that the GitHub MCP server is required for PR reviews.

## Architecture Consistency Check
- Verify changes align with patterns documented in `docs/` directory
- Ensure consistency with existing architectural decisions
- Check for adherence to established design patterns
- Review impact on system architecture

## Before Submitting PR
- Run all tests locally and ensure they pass
- Run linters, formatters, and security scans
- Update documentation if needed
- Add/update tests for new functionality
- Ensure CI/CD pipeline passes
- Self-review your changes for security issues
- Keep PRs small and focused (< 400 lines preferred)
- Check for sensitive data or secrets before committing
- Verify commit messages follow conventional commit format

## Review Guidelines

### 1. Code Quality
- Ensure code is clean, readable, and follows the project's established style guide and coding conventions
- Check for meaningful variable, function, and class names
- Confirm code is modular and avoids duplication
- Verify code follows DRY (Don't Repeat Yourself) principle

#### SonarQube Quality Checks
- **Cognitive Complexity**: Functions should have cognitive complexity within acceptable thresholds (typically ≤15)
  - Refactor complex functions to improve readability and maintainability
  - Break down nested logic into smaller, well-named helper functions
  - Example: "Refactor this function to reduce its Cognitive Complexity from 49 to the 15 allowed"
- **Unused Code**: Remove unused variables, functions, imports, and parameters
  - Check for unused local variables, function parameters, and imports
  - Remove dead code and unreachable statements
  - Example: "Remove the unused local variable 'client'"
- **Exception Handling**: Implement proper exception handling practices
  - Specify exception classes to catch (avoid bare `except:` in Python or empty `catch` in other languages)
  - Either handle exceptions appropriately or reraise them with context
  - Don't suppress exceptions without proper logging or handling
  - Example: "Specify an exception class to catch or reraise the exception"
- **Code Smells**: Address common code smells identified by static analysis
  - Reduce method/function length to improve readability
  - Minimize parameter count (typically ≤5 parameters)
  - Avoid deeply nested control structures
  - Remove duplicated code blocks
- **Code Coverage**: Ensure adequate test coverage as reported by SonarQube
  - Address uncovered or partially covered code paths
  - Focus on critical business logic and edge cases

### 2. Correctness
- Verify the code does what it claims (matches requirements/spec)
- Look for logical errors, edge cases, and off-by-one mistakes
- Ensure error handling is present and appropriate
- Check for proper null/undefined handling

### 3. Security Review
- Check for common security issues (injection, data leaks, unsafe deserialization)
- Verify input validation and output encoding
- Ensure secrets are not hardcoded or exposed
- Check authentication and authorization implementations
- Verify proper error handling without information disclosure
- Review dependency security and versions
- Check for SQL injection, XSS, CSRF vulnerabilities
- Verify proper use of cryptographic functions
- Ensure secure session management

### 4. Testing
- Confirm that new code is covered by unit and/or integration tests
- Tests should follow the project's established testing conventions and best practices
- Ensure tests are meaningful, isolated, and pass reliably
- All new behaviors in the application must have test coverage
- For Python projects, ensure unit tests exclusively use pytest-mock plugin instead of unittest.mock
- Ensure code coverage is at least 80%
- Verify security test cases are included
- Check for edge case testing

### 5. Documentation
- Ensure public APIs, functions, and complex logic are well-documented
- Confirm that README, comments, docstrings and relevant documents are clear and up-to-date
- Check that architectural changes are documented
- Verify API documentation is updated
- Ensure inline comments explain "why" not "what"

### 6. Performance
- Identify unnecessary computations or memory usage
- Suggest optimizations if code is inefficient for its purpose
- Check for N+1 query problems
- Verify proper use of caching where appropriate
- Review database query efficiency

### 7. Maintainability
- Ensure code is easy to extend and maintain
- Check for adherence to SOLID principles and separation of concerns
- Verify proper abstraction levels
- Check for code complexity (cyclomatic complexity)
- Ensure consistent coding patterns

### 8. Dependencies
- Verify that new dependencies are necessary and well-maintained
- Ensure dependency versions are specified with minimum version constraints like `package>=1.2.0`
- Check for security vulnerabilities in dependencies
- Verify license compatibility
- Check for deprecated dependencies

### 9. Compliance
- Confirm code complies with licensing and legal requirements
- Ensure no proprietary or sensitive information is exposed
- Verify GDPR/privacy compliance where applicable
- Check for proper data retention policies

### 10. Feedback Style
- Provide constructive, specific, and actionable feedback
- Highlight strengths as well as areas for improvement, as well as pros/cons of different approaches
- All comments should be in a very friendly tone, and make use of emojis where relevant
- Comments should be inclusive and use "We", instead of "You"
- Distinguish between blocking and non-blocking comments
- Provide code examples for suggestions when helpful

### 11. Formatting Requirements
The entire PR comment should start with:
```
# 🚀 This PR review was conducted by AI 🤖
```

The entire PR comment should end with:
```
Reminder: This PR comment was generated by AI 🤖. Not everything may be accurate - please verify!
```
